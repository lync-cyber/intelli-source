## 问题列表

### [R-001] HIGH: 数据库会话未接驳，所有路由运行时传入 `None`
- **category**: structure
- **root_cause**: self-caused
- **描述**: `api/deps.py` 中 `get_db_session()` 标注 `[ASSUMPTION]` 并 `yield None`。6 个路由文件（`sources.py`、`contents.py`、`tasks.py`、`subscriptions.py`、`search.py`、`llm.py`）各自定义同名 `get_session()` 均 `yield None`，未使用 `DatabaseManager.get_session()`。`main.py` 的生命周期函数 `init_db_pool / init_redis / init_celery` 全为空函数体。`DatabaseManager` 虽完整实现，但从未在 FastAPI 的依赖注入中接入。  
  **影响**: 生产部署时所有写操作（信源 CRUD、任务调度、订阅管理）会因 `None.execute()` 抛出 `AttributeError` 崩溃；当前测试能通过是因为测试直接 mock `Repository`，绕过了会话注入链路。
- **建议**:  
  1. 在 `main.py` lifespan 中实例化 `DatabaseManager` 并存入 `app.state.db`；  
  2. 将 `api/deps.py:get_db_session()` 替换为从 `request.app.state.db.get_session()` yield；  
  3. 删除 6 个路由文件中的重复 `get_session()` 定义，统一 `Depends(get_db_session)`（来自 `api/deps.py`）；  
  4. `init_redis / init_celery` 补充真实初始化逻辑。

### [R-002] HIGH: Agent 工具层全部为占位实现，核心编排路径无法工作
- **category**: completeness
- **root_cause**: self-caused
- **描述**: `agent/tools.py` 中 6 个默认工具（`collect` / `process` / `distribute` / `search` / `get_content_detail` / `summarize_for_user`）的 `execute` 函数均为占位：返回 `{"status": "ok", "tool": "<name>", **kwargs}`，未实际调用 M-002/M-003/M-007/M-008 对应模块。`llm_complete` 元工具同样是占位（`_llm_complete_execute` 不调用 `LLMGateway`）。`AgentRunner.run_flexible()` 依赖这些工具完成 LLM 编排循环，当前等效于"空转"。  
  **影响**: flexible 模式下 Agent 无法真正调用采集/处理/分发/检索模块，也无法发起 LLM 调用，整个 M-006 编排层功能缺失。
- **建议**:  
  1. 在 `AgentToolRegistry.register_defaults()` 或 `_default_tool_defs()` 中注入实际模块依赖（可通过工厂函数或 DI 传入）；  
  2. `llm_complete` 元工具应调用 `LLMGateway.complete()`，`gateway` 实例从 `AgentRunner._llm_gateway` 取得；  
  3. 其余 5 个工具按模块边界接入对应实现（`collect → CollectorRegistry`, `process → PipelineEngine`, `distribute → DistributorBase`, `search → HybridIndex`, `get_content_detail → ContentRepository`）。

### [R-004] MEDIUM: T-060 `GET /api/v1/llm/stats` 为空 stub，API 契约 7 个字段全未实现
- **category**: completeness
- **root_cause**: self-caused
- **描述**: `api/routers/llm.py` 中 `LLMCallLogRepository` 是路由文件内的 stub 类，`get_stats()` 永远返回 `{}`。API-017 要求响应包含：`period`、`total_calls`、`total_tokens`、`total_input_tokens`、`total_output_tokens`、`avg_latency_ms`、`by_model[]`（含 error_rate）、`by_date[]`。`CostTracker.get_stats()` 虽有部分聚合逻辑，但返回字段仅 3 个且缺失 `avg_latency_ms`、`by_model`、`by_date`，亦未聚合 `total_tokens`（sum of input+output）。`storage/repositories/` 无 `llm_call_log.py`，arch M-005 指定的 `LLMStatsAggregator` 组件未实现。  
  **注**: 此为 Sprint 7 T-060 任务，计划中的待实现项，非意外遗漏。
- **建议**: 按 T-060 验收标准实现：在 `storage/repositories/llm_call_log.py` 新增 `LLMCallLogRepository`（含 `get_stats()` 多维聚合 SQL）；更新 `api/routers/llm.py` 接入真实 repo 和 DB 会话；响应结构对齐 API-017 spec。

### [R-005] MEDIUM: `CostTracker.get_stats()` 缺少 latency、by_model、by_date 维度
- **category**: consistency
- **root_cause**: self-caused
- **描述**: `llm/cost_tracker.py:CostTracker.get_stats()` 仅查询 `total_calls / total_input_tokens / total_output_tokens`，缺少：`avg_latency_ms`（`LLMCallLog.latency_ms` 列已存在）、按模型分组（`by_model`）、按日期分组（`by_date`）。API-017 需要这些字段，`LLMCallLog` 数据库表有足够的索引（`ix_llm_call_logs_model`、`ix_llm_call_logs_created_at`），只缺 SQL 聚合逻辑。
- **建议**: 扩展 `CostTracker.get_stats()` 增加：`AVG(latency_ms)` 全局延迟、`GROUP BY model` 子查询、`GROUP BY DATE(created_at)` 子查询；或在新的 `LLMCallLogRepository` 中实现，`CostTracker` 改为纯写入职责。

### [R-008] MEDIUM: 健康检查响应字段缺失（API-018 规范偏离）
- **category**: consistency
- **root_cause**: self-caused
- **描述**: `api/routers/system.py:check_health()` 返回 `{"status": "healthy"}` 固定值。API-018 契约要求：`status`（healthy/degraded/unhealthy）、`version`、`uptime_seconds`、`checks`（含 db/redis/celery 子项）。当前实现缺少 3 个必填字段，也没有实际检查组件存活状态的逻辑。

### [R-011] LOW: `agent/runner.py:155` 在循环体内使用 `import json as _json`
- **category**: convention
- **root_cause**: self-caused
- **描述**: `run_flexible()` 的工具调用循环内有 `import json as _json`，属于函数体内（更严格说：循环体内）导入，违反 PEP 8 顶层导入约定，也有轻微性能影响（每次循环迭代都触发 `sys.modules` 查找）。
