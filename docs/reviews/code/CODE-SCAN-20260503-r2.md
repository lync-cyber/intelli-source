## 新发现问题列表

### [R2-001] HIGH: 应用组合根缺失 — Celery app / AgentRunner / AgentToolRegistry 在生产代码中从未实例化
- **category**: structure
- **root_cause**: self-caused
- **描述**: 全仓 `grep "Celery(" / "celery_app" / "AgentRunner(" / "AgentToolRegistry()"` 在 `src/` 内**零命中**（仅出现在 docstring / 类定义 / 测试 fixture）。具体证据：
  1. `main.py:40 init_celery()` 函数体只有一句 docstring，无 Celery app 创建；`init_redis() / init_db_pool()` 同样为空函数体（r1 R-001 已记录）。
  2. `scheduler/tasks.py:70 CeleryTasks` 类需通过构造器注入 `agent_runner` 与 `pipeline_config`，但**没有任何工厂或模块在生产代码中调用 `CeleryTasks(...)`**——仅 `tests/unit/scheduler/test_tasks.py` 在测试 fixture 中实例化。
  3. `agent/runner.py:19 AgentRunner` 与 `agent/tools.py:29 AgentToolRegistry` 在 `src/` 中无任何 `AgentRunner(...)` / `AgentToolRegistry()` 调用点。`register_defaults()` 与 `register_atomic_tools()` 仅被 `tests/unit/agent/test_tools.py` 与 `tests/unit/agent/test_orchestration.py` 调用。
  4. 没有 `celery_app = Celery(...)` 模块对象、没有 `@shared_task` / `@celery_app.task` 装饰器、没有 worker beat 配置。
- **影响**:
  - 与 r1 R-001（DB session）、r1 R-002（agent 工具占位）合并后，整套 **采集 → 处理 → 编排 → 分发** 链路在生产部署时**全部无法启动**；当前一切"测试通过"的状态完全建立在 fixture 中手工拼装的 mock 对象之上。
  - 即使按 T-075 把 6 个工具的 `_*_execute` 替换为真实模块调用，也没有任何代码会构造 `AgentToolRegistry`、调用 `register_defaults() + register_atomic_tools()`、把它注入到 `AgentRunner`，再把 `AgentRunner` 注入到 `CeleryTasks` —— T-075 本身**不解决**这个问题。
  - PRD F-007（任务调度）/ F-008（Agent 编排）/ arch M-006 当前的"实现状态"实质是单元粒度可测，整体不可运行。
- **建议**:
  1. 新增 `src/intellisource/scheduler/celery_app.py`（或 `app.py`）创建 Celery 实例（broker / backend 从 `config/defaults.yaml` 读取）；在 `main.py:init_celery()` 中 import 这个 module-level singleton。
  2. 新增 `src/intellisource/agent/factory.py`（或在 `scheduler/celery_app.py` 内）提供 `build_agent_runner(session_factory, llm_gateway) -> AgentRunner`：内部构造 `AgentToolRegistry`、调用 `register_defaults()` + `register_atomic_tools()`、注入 `LLMGateway` 与 `ToolDeps`。
  3. `CeleryTasks` 通过 `@celery_app.task` 装饰器注册的具名任务承载，从 `current_app.state.agent_runner` / 全局 lazy singleton 取实例（按 T-072 同时落实 `app.state.db`）。
  4. 集成测试（T-063 / T-071）需新增"端到端 cold start"用例：FastAPI lifespan startup 完成后，调用 `celery_app.send_task("run_pipeline", ...)` 能真实跑通一个 stub pipeline。

### [R2-002] MEDIUM: 上下文压缩存在双实现 — `search/chat_session.py:67` 未跟随 T-058 升级，与 `agent/compaction.py` 策略不一致
- **category**: consistency
- **root_cause**: self-caused
- **描述**: T-058（已 done）将 `agent/compaction.py:compact_messages()` 升级为 token-based 保留 + 结构化 LLM 摘要 + role=tool 优先裁剪 + 失败回退 truncation 的"完整方案"，并补充了 `compaction_summary.txt` 模板。但 `search/chat_session.py:67 compact_context()` 仍是 T-058 之前风格的 string-concat 摘要：
  ```
  summary_text = "Summary of previous conversation: " + "; ".join(summary_parts)
  ```
  且仅按"消息数量除以 10"计算保留窗口（与 T-058 AC-T058-1 显式禁止的"消息数量百分比"策略一致），同时函数 docstring 自承 `[ASSUMPTION] Future versions should integrate an LLM-based compactor`。
- **影响**:
  - Chat 会话（M-008 检索与对话）与 Agent 流（M-006 编排）使用两套质量差距明显的压缩逻辑，相同会话长度下 chat 端的摘要保真度显著低于 agent 端。
  - T-058 的回归保护未覆盖 `search/chat_session.py`，未来修改 `agent/compaction.py` 时该文件可能进一步漂移。
  - 与 arch §M-008 "上下文压缩"职责声明不符（arch 未规定 chat_session 应另有一套压缩，应统一调用 M-006/M-005 的能力）。
- **建议**:
  1. 在 `agent/compaction.py` 暴露一个不依赖 PipelineConfig 的纯函数版本（输入 messages + budget + LLMGateway，输出压缩后的 messages），或将其重构为 `CompactionService` 类。
  2. `search/chat_session.py:compact_context()` 改为委托上述 service，删除本地的 string-concat 实现与 `[ASSUMPTION]` docstring。
  3. 沿 T-058 的同样方式在 `tests/unit/search/test_chat_session.py` 增加压缩回归用例（含 LLM 摘要失败回退）。

### [R2-004] LOW: `storage/models.py` 列模板内部重复 — `id` UUID 主键 + `created_at` / `updated_at` / `started_at` / `finished_at` 模板在 9 张表中各自展开
- **category**: duplication
- **root_cause**: self-caused
- **描述**: jscpd 报告 7 处 `storage/models.py` 内部克隆（`storage/models.py:74-83 / 116-124 / 142-148 / 246-256` 等多对），均为：
  ```python
  id: Mapped[uuid.UUID] = mapped_column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
  ...
  created_at: Mapped[datetime] = mapped_column(TIMESTAMP(timezone=True), nullable=False, server_default=func.now())
  updated_at: Mapped[Optional[datetime]] = mapped_column(TIMESTAMP(timezone=True), onupdate=func.now())
  ```
  以及 `started_at` / `finished_at` 的成对定义。SQLAlchemy 2.0 的 `MappedAsDataclass` + Mixin 模式可显著缩减。
- **影响**: 仅维护成本，不影响运行时；但每次新增表都要 copy-paste 7~10 行模板，是 T-074（TaskChainRepository）后未来扩 schema 时的"无声"返工。
- **建议**: 引入 `class TimestampMixin` 与 `class TimestampedExecutionMixin`（含 started/finished），把列模板压缩到一行 `class Foo(Base, TimestampMixin):`。建议作为 T-074 同 PR 顺手清理（不专项立卡）。

### [R2-005] LOW: vulture 假阳性 — `circuit_breaker.py:19 'names'` 与 `database.py:75 'a'` 不是真实死码
- **category**: dead-code
- **root_cause**: self-caused（工具配置缺失，非源码缺陷）
- **描述**:
  - `llm/circuit_breaker.py:19` 在 `class AsyncRedis(Protocol): async def delete(self, *names: str) -> Any: ...` 中报 `*names` 未使用——这是 typing.Protocol 的方法签名，参数名仅用于文档/类型，**不是真实变量**。
  - `storage/database.py:75 def _closed_creator(*a, **kw): raise RuntimeError(...)` 中的 `*a` / `**kw` 是有意丢弃所有参数的"拒绝创建"闭包，**不能删除**。
- **影响**: 两处都是 vulture 默认阈值下的噪声，会污染未来扫描的 baseline。
- **建议**: 在仓库根 `pyproject.toml`（或 `vulture.toml` / `.vulture_whitelist.py`）加入：
  ```toml
  [tool.vulture]
  ignore_names = ["names", "a", "kw"]
  paths = ["src/intellisource/"]
  min_confidence = 80
  ```
  或在两个变量前加 `_` 前缀（`*_names` / `*_a, **_kw`），让 vulture 自动跳过下划线开头变量。
