## 1. 概述

### 1.1 背景与动机

技术人员和内容创作者日常需要从博客、论文、社交媒体等多种信息源持续获取专业内容，但手动逐一浏览效率低下且容易遗漏关键信息。IntelliSource 旨在提供一套自部署的智能信息聚合与分发系统，通过 LLM 驱动的语义聚类、摘要和精准推送，帮助个人或小团队高效掌握领域动态，将信息获取从"人找信息"转变为"信息找人"。

### 1.2 目标用户

| 角色 | 特征 | 核心诉求 |
|------|------|----------|
| 技术用户（主要） | 具备开发/运维能力，可自行部署和配置系统 | 自动化采集多源信息，获取高质量摘要，减少信息筛选时间 |
| 内容创作者 | 需要持续跟踪特定领域动态，关注热点趋势 | 按学科订阅信息源，获取聚类摘要和趋势分析 |
| 团队管理者 | 小团队负责人，需要为团队统一管理信息订阅 | 集中管理订阅规则，多渠道分发给团队成员 |

### 1.3 成功指标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| 信息采集成功率 | >= 95% | 采集任务成功数 / 总任务数（按日统计） |
| 内容去重准确率 | >= 90% | 抽样检查去重结果的精确率 |
| 摘要生成覆盖率 | 100% 已采集内容 | 生成摘要的内容数 / 采集入库的内容数 |
| 推送到达率 | >= 98% | 成功送达数 / 总推送数（按渠道统计） |
| 用户即时查询响应时间 | P95 < 5s（不含 LLM 等待） | 系统监控指标采集 |
| 系统可用性 | >= 99.5% | 健康检查探针统计 |

## 2. 功能需求

### F-001: 信息源声明式配置

- **用户故事**: 作为技术用户，我希望通过 YAML/JSON 配置文件定义信息源（名称、类型、学科标签、调度策略），以便无需编码即可管理订阅源。
- **验收标准**:
  - [ ] AC-001: 系统支持通过 YAML 或 JSON 文件定义信息源，包含名称、类型、URL、学科标签、采集频率等字段
  - [ ] AC-002: 配置文件变更后无需重启服务即可生效（热加载）
  - [ ] AC-003: 配置文件格式校验失败时，系统拒绝加载并输出明确错误信息，不影响已有配置
  - [ ] AC-004: 支持配置文件版本管理，可回退到历史版本
- **优先级**: P0
- **备注**: 配置格式为系统核心入口，所有信源管理依赖此功能

### F-002: 插件化采集引擎

- **用户故事**: 作为技术用户，我希望系统提供插件化的采集架构，以便新增信息源类型时只需实现统一接口并注册，无需修改核心代码。
- **验收标准**:
  - [ ] AC-005: 系统提供统一的采集器接口定义，新采集器实现该接口即可通过注册机制接入
  - [ ] AC-006: 内置 RSS、通用 API、网页爬虫三种采集适配器
  - [ ] AC-007: 采集数据输出符合统一数据模型（标题、作者、正文、发布时间、来源 URL、元数据）
  - [ ] AC-008: 支持第三方桥接工具作为信源接入（如 RSSHub）
- **优先级**: P0
- **备注**: 依赖 F-001 提供的信息源配置

### F-003: 采集频率自适应与资源隔离

- **用户故事**: 作为技术用户，我希望系统能根据信息源的更新频率自动调整采集间隔，并为不同信源配置独立的代理和限流策略，以便高效利用资源且不触发反爬机制。
- **验收标准**:
  - [ ] AC-009: 系统根据信源历史更新频率动态调整采集间隔（更新频繁则缩短间隔，反之延长）
  - [ ] AC-010: 支持按信息源配置独立的 HTTP 代理
  - [ ] AC-011: 支持按信息源配置独立的请求速率限制（QPS / 并发数）
  - [ ] AC-012: 采集失败时自动重试（至少 3 次），最终失败记录错误日志并触发告警，不阻塞其他任务
- **优先级**: P1
- **备注**: 依赖 F-002 采集引擎基础设施

### F-004: 原子化处理操作与工具注册

- **用户故事**: 作为技术用户，我希望系统将内容处理的各步骤（解析、去重、打标、格式转换、存储、检索等）封装为独立的原子操作，每个操作有统一的输入/输出 Schema，以便灵活组合处理流程并支持外部系统调用。
- **验收标准**:
  - [ ] AC-013: 每个原子操作通过 ToolSpec 注册，定义 name/description/parameters/returns/idempotent/side_effects
  - [ ] AC-014: 支持条件跳过（编排层根据内容类型/标签决定跳过特定操作）
  - [ ] AC-015: 各原子操作独立可替换，实现统一 ToolSpec 接口即可注册
  - [ ] AC-016: 原子操作间通过显式参数传递数据（无隐式共享状态）
  - [ ] AC-017: 支持单条和批量两种调用模式
- **优先级**: P0
- **备注**: 原子操作是系统的核心能力单元，内置 Agent 和外部 Agent 均通过调用这些操作完成业务逻辑；所有原子操作不内置 LLM 调用，LLM 智能由编排层（内置 Agent 或外部 Agent）负责

### F-005: 内置编排 Agent

- **用户故事**: 作为技术用户，我希望系统内置一个 LLM 驱动的编排 Agent，能根据触发信号（定时/手动/消息）自动规划并调用原子操作完成采集-处理-存储-分发全流程，以便实现智能化的自动工作流。
- **验收标准**:
  - [ ] AC-018: 内置 Agent 以 ReAct/Plan-Execute 模式运行，通过 function calling 调用 F-004 定义的原子操作
  - [ ] AC-019: 内置 Agent 附带预定义 Playbook（定时采集、手动采集、即时检索），常见场景走确定性步骤
  - [ ] AC-020: Agent 可利用 LLM 进行语义判断（去重判定、聚类归属、意图理解、摘要生成等）
  - [ ] AC-021: LLM 调用失败时 Agent 自动降级到 Playbook 确定性流程，主流程不中断
  - [ ] AC-022: 每条内容在入库时生成唯一内容指纹，全链路基于指纹幂等处理
- **优先级**: P0
- **备注**: 内置 Agent 是系统独立运行时的智能核心；LLM 不可用时退化为 Playbook 确定性执行，系统仍可运行

### F-006: LLM 服务管理

- **用户故事**: 作为技术用户，我希望系统提供统一的 LLM 调用接口供内置 Agent 使用，包含重试、熔断和成本监控，以便确保 LLM 调用的可靠性和成本可控。
- **验收标准**:
  - [ ] AC-023: 提供统一 LLM 调用接口（基于 litellm），屏蔽不同模型提供商的差异
  - [ ] AC-024: 支持调用失败时自动重试和熔断机制
  - [ ] AC-025: 支持 JSON Mode / Function Calling 确保结构化输出
  - [ ] AC-026: 记录每次调用的模型名称、Token 消耗、延迟、输入输出长度，支持按日/周聚合
  - [ ] AC-027: 敏感词过滤：LLM 调用前后双重检查，命中敏感词标记为需人工审核
- **优先级**: P0
- **备注**: 仅服务内置编排 Agent 的 LLM 调用；外部 Agent 使用自己的 LLM，不经过此模块

### F-007: 任务触发与状态管理

- **用户故事**: 作为技术用户，我希望系统支持定时调度、API 手动触发和消息触发三种模式启动编排 Agent，并提供任务状态追踪，以便灵活控制和监控自动化流程。
- **验收标准**:
  - [ ] AC-028: 支持 Celery Beat 定时触发、API 手动触发、Webhook 消息触发三种模式
  - [ ] AC-029: 定时任务与用户触发任务通过独立优先级队列并行处理，互不阻塞
  - [ ] AC-030: 支持多工作节点并发执行任务
  - [ ] AC-031: 幂等设计：内容指纹去重 + 推送记录表 + 分布式锁，避免重复处理与推送
  - [ ] AC-032: 统一任务状态机：支持暂停、恢复、超时控制
  - [ ] AC-033: Agent 每次执行记录完整的工具调用链（AgentExecutionLog），支持执行回溯
- **优先级**: P0
- **备注**: 触发层负责"何时启动"和"状态追踪"，编排逻辑由内置 Agent (F-005) 或外部 Agent 负责

### F-008: 工作流定义与 Playbook

- **用户故事**: 作为技术用户，我希望能定义自定义工作流（指定信源、处理步骤、分发渠道的组合），并为常见场景提供预定义 Playbook，以便兼顾灵活性与开箱即用。
- **验收标准**:
  - [ ] AC-034: 支持通过 API 定义自定义工作流（采集-处理-分发的灵活组合）
  - [ ] AC-035: 预定义 Playbook：scheduled_collect（定时全量采集处理分发）、manual_collect（指定信源采集）、user_search（即时检索）
  - [ ] AC-036: 工作流支持 Cron 定时执行或仅手动触发
  - [ ] AC-037: Playbook 为纯确定性步骤序列，不依赖 LLM，作为 Agent 降级时的兜底方案
  - [ ] AC-038: 内置 Agent 优先匹配 Playbook 模板，减少不必要的 LLM 推理开销
  - [ ] AC-039: 支持定时调度、轮询检查、用户消息触发三种触发模式
- **优先级**: P0
- **备注**: Playbook 与工作流的关系：Playbook 是系统内置的常见流程模板，工作流是用户自定义的流程定义，两者共用原子操作层

### F-009: 多渠道分发

- **用户故事**: 作为订阅用户，我希望通过微信公众号、企业微信或邮件接收订阅内容推送，以便在常用渠道获取最新信息。
- **验收标准**:
  - [ ] AC-040: 支持微信公众号推送（模板消息/图文消息）
  - [ ] AC-041: 支持企业微信应用消息推送
  - [ ] AC-042: 支持邮件推送（HTML 格式）
  - [ ] AC-043: 基于用户订阅规则（关键词、学科标签）精准匹配推送内容
  - [ ] AC-044: 同一内容对同一用户同一渠道不重复推送
  - [ ] AC-045: 推送失败自动重试，记录推送历史
  - [ ] AC-046: 支持推送频率控制与免打扰时段配置
- **优先级**: P0
- **备注**: v1 优先支持微信公众号/企业微信 + 邮件；原始需求中提及的 Telegram 渠道在 v1 不纳入，原因为目标用户以国内渠道为主，Telegram 在国内使用受限 [ASSUMPTION]。其他渠道（钉钉、Webhook、Telegram 等）通过统一分发接口预留扩展点，后续迭代接入

### F-010: 推送内容智能优化

- **用户故事**: 作为订阅用户，我希望推送的内容经过智能重排序并附带引导语，以便更快把握信息重点。
- **验收标准**:
  - [ ] AC-047: 内置 Agent 在分发前可调用 LLM 对内容进行重排序（按相关性/重要性）
  - [ ] AC-048: 内置 Agent 为推送内容生成简短引导语/摘要
  - [ ] AC-049: LLM 不可用时 Agent 降级到 Playbook：默认时间排序 + 无引导语
- **优先级**: P2
- **备注**: 推送优化由内置 Agent 在编排流程中决定是否执行；外部 Agent 可自行在调用 push 原子操作前完成优化

### F-011: 消息指令式即时检索

- **用户故事**: 作为订阅用户，我希望通过微信/企业微信发送消息指令触发即时检索，系统理解我的意图后返回相关摘要，以便满足临时信息需求。
- **验收标准**:
  - [ ] AC-050: 用户通过消息渠道发送自然语言指令后，内置 Agent 调用 LLM 理解检索意图
  - [ ] AC-051: 基于理解的意图调用 search_hybrid 原子操作执行混合检索，返回相关结果
  - [ ] AC-052: 内置 Agent 对检索结果生成摘要后异步回调返回给用户，不阻塞消息通道
  - [ ] AC-053: 支持多轮对话上下文保持 [ASSUMPTION: 保持最近 5 轮对话上下文]
- **优先级**: P1
- **备注**: 即时检索流程：Webhook 接收消息 → 触发内置 Agent(user_search Playbook) → Agent 调用原子操作检索+摘要 → 异步回调返回

### F-012: 存储与混合检索

- **用户故事**: 作为技术用户，我希望系统将结构化数据存入关系型数据库、内容向量存入向量数据库，并提供混合检索接口，以便支持高精度的内容查找。
- **验收标准**:
  - [ ] AC-054: 结构化数据（信源配置、任务状态、推送记录等）存入关系型数据库
  - [ ] AC-055: 内容正文和摘要的向量表示存入向量数据库，支持语义检索
  - [ ] AC-056: 提供混合检索接口，支持关键词与语义向量联合查询，结果按相关性排序
- **优先级**: P0
- **备注**: 为 F-005 语义去重、F-011 即时检索提供存储和检索基础

### F-013: 可观测性

- **用户故事**: 作为技术用户，我希望系统提供结构化日志、指标监控和分布式链路追踪，以便快速定位和排查问题。
- **验收标准**:
  - [ ] AC-057: 所有日志包含任务标识、处理阶段、耗时等结构化字段
  - [ ] AC-058: 采集成功率、延迟、队列长度、LLM Token 用量等关键指标可通过监控系统查看
  - [ ] AC-059: 全链路请求可通过 Trace ID 串联，从采集到分发完整追踪
  - [ ] AC-060: 健康检查端点可用，关键指标异常时自动触发告警
- **优先级**: P1
- **备注**: 自部署场景下可观测性对运维至关重要

### F-014: RESTful API 与 CLI

- **用户故事**: 作为技术用户，我希望通过 RESTful API 和命令行工具管理信源、触发任务、查询状态和定义工作流，以便实现自动化操作和与其他系统集成。
- **验收标准**:
  - [ ] AC-061: API 支持信源的 CRUD 操作（创建、查询、更新、删除）
  - [ ] AC-062: API 支持手动触发采集任务和查询任务执行状态
  - [ ] AC-063: API 支持定义和执行自定义工作流（采集-处理-分发的灵活组合）
  - [ ] AC-064: 提供 CLI 工具封装常用 API 操作
  - [ ] AC-065: API 文档自动生成（OpenAPI/Swagger 格式）
- **优先级**: P1
- **备注**: API 是系统的外部集成入口，也为 F-011 消息指令式检索提供后端支撑

### F-015: 原子操作外部暴露（MCP + API）

- **用户故事**: 作为外部 AI Agent 的开发者，我希望 IntelliSource 的所有原子操作通过 MCP (Model Context Protocol) 和 RESTful API 双协议暴露，以便我的 Agent 可以直接编排这些操作完成自定义流程，无需依赖系统内置的 LLM 处理逻辑。
- **验收标准**:
  - [ ] AC-066: 所有 F-004 注册的原子操作同时通过 FastAPI 端点 (`/api/v1/tools/{tool_name}`) 和 MCP Tool 协议暴露
  - [ ] AC-067: MCP Server 从 ToolSpec 注册表自动生成工具定义（name/description/input_schema），无需手动维护
  - [ ] AC-068: 外部 Agent 可通过原子操作完成完整的采集-处理-存储-分发流程，不依赖内置 Agent
  - [ ] AC-069: embedding 向量由外部 Agent 生成并通过 store_processed 操作传入，系统不强制使用内部 embedding 模型
  - [ ] AC-070: 原子操作的 MCP 调用与 API 调用共享相同的认证、限流和可观测性基础设施
- **优先级**: P1
- **备注**: 这是系统通用性的关键特性——使 IntelliSource 既能独立运行（内置 Agent），又能作为外部智能体的工具集被集成

## 3. 非功能需求

### 3.1 性能

| 场景 | 指标 | 目标值 |
|------|------|--------|
| 用户即时检索请求 | 响应时间 P95 | < 5s（不含 LLM 调用等待） |
| 配置热加载 | 生效延迟 | < 10s |
| 单节点采集吞吐 | 并发采集任务数 | >= 20 [ASSUMPTION] |
| Agent Playbook 降级 | LLM 不可用到 Playbook 接管 | < 500ms |
| 推送分发 | 单批次推送完成时间 | < 30s（100 条消息） [ASSUMPTION] |

### 3.2 安全

- API 访问需认证授权（API Key 或 Token 机制） [ASSUMPTION: v1 采用 API Key 认证，不引入完整的用户体系]
- LLM API 密钥等敏感配置支持加密存储或环境变量注入，不允许明文存储在配置文件中
- 采集的内容数据仅存储在本地部署环境中，不上传至外部服务（LLM 调用除外）
- 敏感词过滤与合规检查（见 F-006 AC-026）

### 3.3 兼容性

- 部署环境: Linux (Ubuntu 20.04+/CentOS 8+)、macOS、Windows (WSL2) [ASSUMPTION]
- 容器化部署: 支持 Docker / Docker Compose 一键部署
- 消息渠道: 微信公众号、企业微信、SMTP 邮件（v1 范围）
- 数据库: 关系型数据库和向量数据库（具体选型由架构设计阶段确定）

## 4. 约束与假设

- **约束**:
  - 系统为自部署工具，不提供 SaaS 托管服务
  - v1 版本消息渠道限定为微信公众号、企业微信和邮件，其他渠道预留接口
  - LLM 调用依赖外部服务（OpenAI / 国内模型提供商），网络连通性为前提
  - 全功能交付（F-001 至 F-015），按 Sprint 迭代完成

- **假设**:
  - [ASSUMPTION] 用户具备基本的服务器运维能力，可自行完成 Docker 部署和配置
  - [ASSUMPTION] 目标用户数量级为个人或 10 人以内小团队，不考虑多租户隔离
  - [ASSUMPTION] v1 不提供 Web 管理界面，通过配置文件 + API + CLI 管理
  - [ASSUMPTION] LLM 成本由用户自行承担（使用自己的 API Key）
  - [ASSUMPTION] 向量数据库初始数据规模 < 100 万条文档
  - [ASSUMPTION] v1 不实现自动数据清理，用户需手动管理存储空间；数据保留策略与自动清理计划在后续版本实现（对应原始需求 F10）

## 5. 术语表

| 术语 | 定义 |
|------|------|
| 信息源 (Source) | 可采集内容的外部数据来源，如 RSS 订阅、API 接口、网页等 |
| 采集器 (Collector) | 负责从特定类型信息源抓取内容的插件组件 |
| 处理管道 (Pipeline) | [已废弃] 旧版概念，v1 架构中由"原子操作 + Agent 编排"替代 |
| 处理器 (Processor) | [已废弃] 旧版概念，v1 架构中由"原子操作 (Atomic Tool)"替代 |
| 学科标签 (Topic Tag) | 用于分类信息源和内容的领域标识，如"AI"、"前端"等 |
| 内容指纹 (Fingerprint) | 基于内容特征生成的唯一标识，用于全链路幂等去重 |
| 语义去重 | 利用向量相似度和 LLM 判定，识别语义相同但表述不同的内容 |
| 聚类 (Clustering) | 将同一事件或主题的多源内容归为一组 |
| 综合简报 (Digest) | 对同主题多篇内容的聚合摘要，提炼各源要点 |
| 分发 (Distribution) | 将处理后的内容通过配置的渠道推送给订阅用户 |
| LLM 网关 (LLM Gateway) | 统一管理多 LLM 提供商调用的中间层，提供重试、熔断、降级能力 |
| 降级 (Fallback) | LLM 调用失败时自动切换到传统处理逻辑的机制 |
| 熔断 (Circuit Breaker) | 当 LLM 调用连续失败达到阈值时，暂时停止调用以保护系统 |
| 幂等 (Idempotent) | 同一操作执行多次与执行一次效果相同，避免重复处理 |
| 热加载 (Hot Reload) | 配置变更后无需重启服务即自动生效 |
| 混合检索 (Hybrid Search) | 结合关键词匹配和向量语义检索的查询方式 |
| 原子操作 (Atomic Tool) | 系统功能的最小执行单元，有统一 I/O Schema，不内置 LLM，可被内置/外部 Agent 调用 |
| 内置编排 Agent | 系统内置的 LLM 驱动编排器，通过调用原子操作完成业务流程 |
| Playbook | 预定义的确定性步骤序列，作为 Agent 的快速路径和 LLM 不可用时的降级方案 |
| ToolSpec | 原子操作的统一描述协议，定义 name/parameters/returns/idempotent/side_effects |
| MCP (Model Context Protocol) | 开放协议，允许外部 AI Agent 发现和调用系统暴露的工具 |
